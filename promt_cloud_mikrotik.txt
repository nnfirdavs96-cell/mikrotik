Ты опытный Senior Python Backend Developer, Linux Engineer, MikroTik RouterOS API Specialist и разработчик Captive Portal систем.

Нужно создать полноценную MVP-программу для управления доступом клиентов к интернету через главный MikroTik.

Проект должен быть написан на Python 3 и работать на Linux сервере: Ubuntu Server или CentOS.

Главная задача:
Создать систему, где клиент после подключения к Wi-Fi сам проходит регистрацию через mini web-интерфейс / captive portal, вводит номер телефона, получает SMS-код, подтверждает номер, выбирает тариф, оплачивает через электронный кошелёк, после чего интернет автоматически активируется на его устройстве через главный MikroTik.

Также должна быть web-панель администратора, где можно управлять MikroTik, клиентами, тарифами, платежами, логами и вручную активировать/деактивировать клиентов.

Система должна состоять из трёх основных частей:

1. Admin Panel
Для администратора:
- добавление MikroTik;
- проверка подключения к MikroTik;
- управление клиентами;
- управление тарифами;
- просмотр платежей;
- просмотр SMS/OTP логов;
- просмотр access logs;
- ручная активация/деактивация клиента;
- синхронизация базы с MikroTik.

2. Client Portal / Captive Portal
Для клиента:
- подключился к Wi-Fi;
- открыл mini web-интерфейс;
- ввёл номер телефона;
- получил SMS-код;
- подтвердил SMS-код;
- выбрал тариф;
- оплатил тариф;
- получил интернет автоматически.

3. MikroTik API Service
Сервис, который:
- подключается к MikroTik через RouterOS API;
- получает IP и MAC клиента из DHCP leases;
- добавляет IP клиента в address-list allowed_clients;
- удаляет IP клиента из allowed_clients;
- синхронизирует базу с MikroTik.

ОБЩАЯ ЛОГИКА РАБОТЫ

Есть:
1. Главный MikroTik.
2. Несколько Wi-Fi точек доступа / AP.
3. Клиенты подключаются к Wi-Fi.
4. MikroTik выдаёт клиенту IP через DHCP.
5. Клиент пока не имеет доступа к интернету.
6. Клиент открывает mini portal.
7. Backend определяет IP клиента через request.client.host.
8. Backend через MikroTik RouterOS API находит DHCP lease клиента.
9. Из DHCP lease берётся:
   - IP-адрес клиента;
   - MAC-адрес клиента;
   - hostname, если есть.
10. Клиент сам вводит номер телефона.
11. Backend отправляет SMS-код.
12. Клиент вводит SMS-код.
13. После подтверждения клиент выбирает тариф.
14. Клиент оплачивает тариф через электронный кошелёк.
15. После успешной оплаты backend активирует клиента.
16. IP клиента добавляется в MikroTik address-list allowed_clients.
17. Клиент получает интернет.
18. Когда срок тарифа заканчивается, система должна деактивировать клиента и удалить IP из allowed_clients.

ВАЖНО:
Клиент не должен сам вводить MAC-адрес и IP-адрес.
MAC и IP должны автоматически определяться через MikroTik DHCP lease после подключения клиента к Wi-Fi.

Для MVP реальные SMS и реальную оплату можно сделать через mock provider, но архитектура должна быть готова для подключения настоящего SMS API и настоящей платёжной системы.

ВАЖНОЕ ТРЕБОВАНИЕ ПО MIKROTIK

Управление главным MikroTik должно выполняться только через MikroTik RouterOS API.

Нельзя использовать:
- SSH;
- telnet;
- WinBox;
- ручной ввод команд;
- выполнение CLI-команд через shell.

Python-программа должна подключаться к MikroTik через RouterOS API и автоматически выполнять все действия.

Использовать порт:
- 8728 для обычного API;
- 8729 для API-SSL, если включён SSL.

Для работы с MikroTik использовать Python-библиотеку:
- librouteros
или
- routeros-api.

Лучше использовать librouteros, если она стабильнее для RouterOS v6/v7.

Основные разделы MikroTik, с которыми должна работать программа:

1. DHCP leases:

/ip/dhcp-server/lease

Нужно получать:
- address;
- mac-address;
- host-name;
- status;
- dynamic;
- comment.

2. Firewall address-list:

/ip/firewall/address-list

Основная address-list:

allowed_clients

Логика доступа:
- если клиент активирован, его IP добавляется в address-list allowed_clients;
- если клиент деактивирован, его IP удаляется из address-list allowed_clients.

Комментарий в MikroTik address-list должен быть таким:

wifi-client | phone=<phone> | mac=<mac_address> | client_id=<client_id>

Пример:

wifi-client | phone=+992900000000 | mac=AA:BB:CC:DD:EE:FF | client_id=1

MikroTik firewall-логика:
В MikroTik должны быть правила:
1. Разрешить клиентам из allowed_clients выход в интернет.
2. Заблокировать остальных клиентов из гостевой Wi-Fi сети.
3. Разрешить DHCP.
4. Разрешить DNS.
5. Разрешить доступ к серверу portal.
6. Разрешить доступ к платёжной системе и SMS provider, если это нужно.

Пример гостевой сети:

192.168.50.0/24

Пример backend portal server:

192.168.50.2 или 192.168.0.100

В README нужно добавить пример правил MikroTik RouterOS.

Пример:

/ip firewall filter
add chain=forward src-address-list=allowed_clients action=accept comment="Allow paid WiFi clients"
add chain=forward src-address=192.168.50.0/24 dst-address=<BACKEND_SERVER_IP> action=accept comment="Allow access to portal server"
add chain=forward src-address=192.168.50.0/24 protocol=udp dst-port=53 action=accept comment="Allow DNS for guest clients"
add chain=forward src-address=192.168.50.0/24 protocol=udp dst-port=67,68 action=accept comment="Allow DHCP for guest clients"
add chain=forward src-address=192.168.50.0/24 action=drop comment="Block unpaid WiFi clients"

/ip firewall nat
add chain=srcnat src-address=192.168.50.0/24 out-interface-list=WAN action=masquerade comment="NAT guest WiFi"

Важно:
В README обязательно добавить предупреждение:
- перед применением firewall сделать backup MikroTik;
- сначала тестировать на отдельной guest Wi-Fi сети;
- не применять правила на основной офисной сети;
- сначала добавить разрешающие правила для админской сети;
- не применять drop rule без проверки;
- API-порт MikroTik нельзя открывать в интернет;
- разрешить доступ к API только с IP сервера, где работает программа;
- для production лучше использовать API-SSL 8729.

НАСТРОЙКА MIKROTIK API

В README добавить инструкцию:

Включить API:

/ip service enable api
/ip service set api port=8728

Для SSL:

/ip service enable api-ssl
/ip service set api-ssl port=8729

Создать отдельного пользователя для API:

/user group add name=api-policy policy=read,write,api,test
/user add name=api_user group=api-policy password=strong_password

Ограничить доступ к API только с IP backend-сервера:

/ip firewall filter
add chain=input src-address=<SERVER_IP> protocol=tcp dst-port=8728 action=accept comment="Allow MikroTik API from backend server"
add chain=input protocol=tcp dst-port=8728 action=drop comment="Block MikroTik API from others"

MIKROTIK MODULE

Создать класс MikroTikAPIClient.

Файл:

app/mikrotik/client.py

Методы:

1. connect()
Подключение к MikroTik через RouterOS API.

2. check_connection()
Проверка доступности MikroTik.

3. get_system_resource()
Получить /system/resource для проверки.

4. get_dhcp_leases()
Получает список DHCP leases из MikroTik:
- IP address;
- MAC address;
- hostname;
- status;
- dynamic/static;
- comment.

5. find_lease_by_ip(ip_address)
Находит DHCP lease по IP-адресу.

6. find_lease_by_mac(mac_address)
Находит DHCP lease по MAC-адресу.

7. get_online_clients()
Получает список клиентов, которые сейчас получили IP через DHCP.

8. add_ip_to_allowed_list(ip_address, phone, mac_address, client_id, list_name="allowed_clients")
Добавить IP клиента в address-list.

9. remove_ip_from_allowed_list(ip_address, list_name="allowed_clients")
Удалить IP клиента из address-list.

10. is_ip_in_allowed_list(ip_address, list_name="allowed_clients")
Проверить, есть ли IP в address-list.

11. get_allowed_clients(list_name="allowed_clients")
Получить список IP в address-list.

12. sync_allowed_clients(active_clients, list_name="allowed_clients")
Синхронизировать активных клиентов.

Правила:
- не создавать дубликат, если IP уже есть;
- если IP отсутствует при удалении, не считать это критической ошибкой;
- все ошибки возвращать понятно;
- программа не должна падать при недоступности MikroTik;
- все ошибки записывать в access_logs.

ОПРЕДЕЛЕНИЕ IP И MAC КЛИЕНТА

Когда клиент открывает portal, backend должен:

1. Получить IP клиента:

request.client.host

2. Через MikroTik API найти DHCP lease:

/ip/dhcp-server/lease

3. Найти lease, где:

address = client_ip

4. Из lease получить:
- mac-address;
- address;
- host-name;
- status.

5. Эти данные связать с регистрацией клиента.

Если DHCP lease не найден:
Показать ошибку:

"Не удалось определить устройство. Переподключитесь к Wi-Fi и попробуйте снова."

CLIENT PORTAL FLOW

Реализовать полный flow:

1. Клиент подключается к Wi-Fi.
2. MikroTik DHCP выдаёт IP.
3. Клиент открывает любой сайт или вручную открывает portal.
4. MikroTik firewall/NAT должен перенаправить клиента без доступа на portal или клиент открывает portal вручную.
5. Клиент открывает portal.
6. Backend определяет IP клиента.
7. Backend через MikroTik API получает MAC из DHCP lease.
8. Клиент вводит номер телефона.
9. Backend отправляет SMS-код.
10. Клиент подтверждает SMS.
11. Клиент выбирает тариф.
12. Backend создаёт payment.
13. Клиент оплачивает.
14. Payment provider сообщает backend об успешной оплате.
15. Backend активирует клиента.
16. Backend добавляет IP клиента в MikroTik allowed_clients.
17. Клиент получает интернет.

ОБРАБОТКА ОШИБОК

Если MikroTik недоступен:
- программа не должна падать;
- web-панель должна показать ошибку;
- REST API должен вернуть понятный JSON;
- ошибка должна записаться в access_logs.

Пример ответа:

{
  "success": false,
  "message": "MikroTik API connection failed",
  "details": "Connection timeout"
}

Если DHCP lease не найден:
- показать в portal понятную ошибку:
"Не удалось определить устройство. Переподключитесь к Wi-Fi и попробуйте снова."

Если клиент уже активирован:
- не добавлять дубликат в MikroTik;
- вернуть success=true;
- написать message: "Client already active".

Если IP уже есть в address-list:
- не добавлять второй раз;
- обновить статус в базе;
- записать лог.

Если IP нет при деактивации:
- считать операцию успешной;
- записать лог: "IP was not found in allowed_clients".

Если SMS не отправлен:
- записать ошибку в sms_logs;
- показать клиенту сообщение.

Если оплата не прошла:
- payment.status = failed;
- показать клиенту страницу payment-failed.
